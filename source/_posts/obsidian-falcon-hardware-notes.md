# 概述
本笔记主要注重于了解关于falcon硬件实现相关的内容，主要将其与UEC的硬件实现相对比，用于了解这些协议设计的区别之处。
# Falcon的存在形式
## 1. 物理形态
- Falcon核心被设计成一个IP核，被放入到网卡芯片之中，Falcon 已经集成到了 **Intel E2100 IPU** 中，支持 **200Gbps** 的速度，并且已经扩展设计支持 **400Gbps** 和 **800Gbps**。
- 并未采用FPGA形式，理由是FPGA虽然灵活，但是太贵且在高速下性能不良，使用ASIC可以满足成本和性能要求。**ASIC（专用集成电路）** 虽然开发周期长、成本高，但在量产后单个成本低，且能提供极致的性能和功耗比。

## 3. 集成时的挑战
1. 协议适配：
	1. 将RDMA模块的操作更换为falcon的事物格式，类似于UEC的SES层。
	2. Falcon 需要极高精度的发包控制（Pacing），这需要一个定时器。
2.  **时间轮 (Timing Wheel) —— 独立的节拍器：**
	3. Google 并没有把这个定时器硬塞进 Falcon 内部，而是作为一个**独立的共享模块**。这样其他需要定时功能的模块也能复用它，或者 Falcon 也可以直接调用网卡里已有的定时轮。
	4. Falcon 需要极高精度的发包控制（Pacing），这需要一个定时器.
3. **时间戳 (Timestamps) —— 必须靠前站：** 
	- Falcon 需要极其精准的 RTT（往返时间）来控制拥塞。.
	- 如果由 Falcon 自己打时间戳，那么数据从 Falcon 走到物理网口（MAC/PHY）这段路程的内部延迟就会干扰测量。
	- **解决方案：** 时间戳的记录工作被外包给了**靠近物理网口的“内联加密模块” (Inline Encryption / PSP / IP-SEC)**.
4. 内部构造：收发分离
	* 发送流水线：接受请求，组装包头，查表，计算窗口并发送
	* 接受流水线：解析进来的包，检查ACK，更新状态并剥离数据。
	* 共享控制表
## 4. 一些小巧思
1. 使用巨大的SRAM来加快速度：sram本身一般作为处理器内的Cache使用。容量小的同时占地大。一般来说不会用来存储数据包载荷，但是Falcon选择使用sram进行数据包缓存。Falcon 必须能暂存乱序到达的包。如果每次存取这些包都要去片外 DRAM 绕一圈，延迟就太高了，根本跑不满 200G/400G 的速度。Google 算了一笔账。随着制程工艺进步（7nm, 5nm, 3nm），存储单元变小了。
	- 要存下一个微秒级 RTT 内的数据（BDP），大概需要 1~5MB 的空间。
	- 这在现代芯片上只占 **不到 1% 的面积**。用 1% 的面积换取极致的抗乱序能力，血赚。
2. 为了缓存命中率进行堆料。
	* 仓库级计算（Warehouse-scale）的一个特点是**并发连接数极高**（O(100K) 级别）。
	* 芯片内的“连接状态缓存（Connection Cache）”存不下这么多连接的状态。一旦你要处理的连接不在缓存里（Cache Miss），芯片就要去慢速内存里找，导致性能瞬间暴跌（性能悬崖）。
	* 解决方案：加缓存，增加访存速度
![[Pasted image 20251225105247.png]]
3. 将异常处理放进硬件中：一般来说，传统的网络架构硬件总是预期数据能够无损失无错误的进行传输，将错误处理分配到软件进行处理，这种预期在很多时候是无法成立的，在高性能和大规模的网络中，丢包以及乱序等问题是非常常见的情况 ，假如全部交给软件处理，会导致片上处理器瞬间超过处理上线，导致性能下降。
	- falcon将复杂的错误处理直接做到片上，虽然在开发阶段需要花费更多的事件和精力，但是事实上在应用中能够极大的减少错误处理所需要的资源。
	- falcon硬件能够处理的错误包括丢包、乱序、超时等。
## 5.   **实现与扩展 Falcon 自适应引擎**
所谓的**Falcon 自适应引擎** 就是数据面与硬件面的沟通渠道，我们知道，硬件与软件需要进行沟通，通过这种方式来进行调度与控制。例如控制发送窗口，控制丢包处理等等，这些内容需要获取来自数据面的信息并对数据面进行控制，falcon的这一套机制是使用**共享内存队列 (Shared Memory Queues)** 来对话的。
	为了效率，Falcon 精确切割了任务（参考论文中的 Table 3）：
- **硬件 (PDL) 负责“机制”：**
    - 测量 RTT（时间戳）。
    - 监控缓存水位。
    - 执行窗口限制。
- **软件 (FAE) 负责“策略”：**
    - **计算：** 计算 fcwnd（拥塞窗口）、alpha（隔离系数）、重传超时时间（RTO）。
    - **决策：** 决定要不要切换多路径的 Flow Label。
	最核心的挑战：状态管理与缓存未命中 (The Scaling Challenge)
这是这一节最精彩的部分，讲述了 Google 工程师在设计 FAE 时的**“踩坑”与“进化”**过程。

**问题背景：**  
每个连接都有自己的“状态”（比如当前的窗口大小、历史延迟数据等）。FAE 每次处理事件时，都需要读取这些状态。

**方案 A：无状态 FAE (Stateless FAE) —— 也就是把状态存在硬件里**
- **做法：** PDL 硬件自己保管状态。每次发 Event 给 FAE 时，顺便把状态数据“夹带”在 Event 消息里发过去。FAE 算完后，把新状态夹带在 Response 里还给 PDL。
- **优点：** FAE 不需要去内存里找数据，总是能命中 CPU 缓存（Hot Cache），处理极快。
- **缺点：** **带不动了。** 硬件 Event 的消息长度是有限的，如果算法越来越复杂（需要记录几十个历史数据），消息包就塞不下了。
**方案 B：有状态 FAE (Stateful FAE) —— 把状态存在软件内存里**
- **做法：** 状态数据存在 FAE 能够访问的内存里。Event 消息里只给一个 ID，FAE 自己去内存里查状态。
- **优点：** 空间大，想存多少存多少，支持复杂算法。
- **新问题（Cache Miss）：** 当连接数达到 10 万级别时，CPU 的 L1/L2/L3 缓存存不下所有连接的状态。FAE 每次去内存里捞数据，都会发生 **Cache Miss（缓存未命中）**，导致 CPU 停顿等待，处理速度变慢。
**最终方案：有状态 + 预取技术 (Stateful with Prefetching)**  
为了解决方案 B 的慢，Falcon 引入了**软件流水线预取**：
- **原理：** 就像一个一边炒菜一边看菜单的厨师。
- **操作：** 当 FAE 的 CPU 正在处理“事件 A”的时候，它会利用空闲的内存带宽，**偷看队列里后面的“事件 B、事件 C”**，并提前发出指令，把 B 和 C 需要的状态数据从内存里拉取到 CPU 缓存中。
- **效果：** 等处理完 A，转头处理 B 时，B 的数据已经在缓存里候着了。
- **战绩：** 这种优化消除了内存延迟的影响，使得即使在极高并发下，FAE 依然能保持极高的处理效率（参考论文图 22b）。
