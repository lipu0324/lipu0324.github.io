UEC 的传输路径可以从 PDS，也就是 Packet Delivery Sublayer 开始理解。PDS 负责在语义层和底层网络之间维护包传输、顺序、确认、重传和连接资源。

## 分层位置

一个简化的 UEC 数据路径可以分成：

1. `libfabric API`：向上承接应用和通信库。
2. `SES`：语义层，把上层操作映射成可传输请求。
3. `PDS`：包传输层，管理 PDC、PSN、ACK/NACK 等机制。
4. `CCC`：拥塞控制。
5. `TSS`：安全相关处理。
6. IP/UDP/Ethernet：承载 UEC 包。

其中 PDS 是连接 SES 语义和网络数据包的核心层。

## 一次请求的基本路径

一次 PDS 数据传输可以按下面的顺序理解：

1. 发起端 SES 生成请求。
2. 发起端 PDS 将请求封装为 PDS Request。
3. 网络把 PDS Request 送到目标端。
4. 目标端 PDS 解包并交给目标 SES。
5. 目标 SES 处理请求并产生响应。
6. 响应通过 PDS ACK 或新的 PDS Request 返回发起端。
7. 发起端 PDS 将结果交付给本地 SES。

小响应可以随 ACK 返回，减少一次独立发包；大响应则更可能作为新的数据请求走回程路径，并参与拥塞控制。

## PSN的作用

PSN 是 Packet Sequence Number，用来标记包在序列中的位置。它承担几个关键职责：

- 判断包是否按预期到达。
- 支持乱序后的重排。
- 支持 ACK/NACK 和重传。
- 帮助接收端识别旧包、重复包或非法包。

PDS 的可靠性并不是简单照搬 TCP，而是在面向高性能 fabric 的上下文中重新设计顺序和恢复机制。

## PDC握手

PDC 可以理解为一段 PDS 层的传输上下文。建立 PDC 时，常见字段包括：

| 字段 | 含义 |
|---|---|
| `SYN` | 表示正在建立连接 |
| `SPDCID` | 源端 PDC ID |
| `DPDCID` | 目的端 PDC ID |
| `PSN` | 当前包序列号 |
| `OFFSET` | 用于推导起始 PSN 的偏移 |
| `ACK` | 确认某个序列状态 |

握手流程可以概括为：

1. 发起端发送带 `SYN` 的包。
2. 目标端收到 `SYN` 后分配本地 PDC 标识，并推导起始 PSN。
3. 目标端返回 ACK，携带确认信息。
4. 发起端收到 ACK 后，后续包不再携带 `SYN`。
5. PDC 进入正常传输状态。

## 资源不足时的处理

PDS 还需要处理 out-of-resource 情况。比如 PDC 表项已满时，管理层不能简单丢弃所有新请求，而需要：

- 把请求放入等待队列。
- 检查是否存在可关闭或超时的 PDC。
- 触发 PDC close。
- 资源回收后重新处理等待请求。

这部分逻辑很适合抽象成管理状态机，而不是散落在发送和接收路径里。

## 状态机视角

可以把 PDS Manager 的行为分为三类：

- TX 请求：查找或创建 PDC，然后把包转发给对应 PDC。
- RX 包：校验包、查找目标 PDC，必要时建立新连接。
- 资源管理：处理超时、关闭和等待队列。

这种拆分能让协议状态、资源状态和数据路径更清楚，也更适合后续 HLS 或硬件流水线实现。

## 小结

PDS 层关注的不是单个包能否被发出去，而是一组包如何在高性能网络中被可靠、可控地发送。PDC 提供传输上下文，PSN 提供顺序语义，ACK/NACK 提供恢复机制，资源管理则保证系统在高并发下不会失控。
